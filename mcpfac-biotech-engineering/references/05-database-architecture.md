# Volume 5 — Database Architecture (PostgreSQL, Prisma, Supabase, Data Models)

## Database Mission

The database is the single source of truth: highly normalized, fast,
secure, scalable, extensible, auditable, easy to maintain. Every table has
a clear purpose; avoid duplicated data — use relationships instead of
repeated values.

## Database Stack

- Database: Supabase PostgreSQL
- ORM: Prisma
- Migrations: Prisma Migrate
- Storage: Supabase Storage
- Auth: Supabase Auth

## Design Principles

Third Normal Form (3NF), referential integrity, foreign keys, indexes, soft
deletes, `createdAt`/`updatedAt`, UUID primary keys, optimistic locking
(future), audit logging.

Every table includes: `id`, `createdAt`, `updatedAt`, `deletedAt`
(nullable), `createdBy` (nullable), `updatedBy` (nullable). Use UUIDs, never
auto-increment integers.

## Database Domains

Authentication, Users, Products, Categories, Orders, Cart, Quotes,
Invoices, Downloads, Documents, Research, Blog, Support, Messaging,
Analytics, System, Settings, Media, Notifications.

## Authentication

Supabase Auth manages authentication, email verification, password reset,
session management, JWT tokens. The application database stores profile
information only.

## Users Domain

Tables: `profiles`, `roles`, `permissions`, `role_permissions`,
`user_roles`, `addresses`, `contact_preferences`, `user_sessions`,
`user_devices`, `activity_logs`, `notification_preferences`. A user may
have multiple roles.

## Customer Domain

Tables: `customers`, `customer_groups`, `customer_notes`,
`customer_documents`, `customer_tags`, `customer_preferences`,
`pricing_groups`. Customer groups: Retail, Research, Distributor,
Wholesale, VIP, Institution, University, Laboratory.

## Product Domain

Tables: `products`, `product_categories`, `product_images`,
`product_files`, `product_variants`, `product_specifications`,
`product_attributes`, `product_prices`, `product_inventory`,
`product_tags`, `product_reviews`, `product_questions`,
`product_answers`, `product_related`, `product_frequently_bought`,
`product_recently_viewed`, `product_downloads`, `product_certificates`.
Every product supports unlimited images, downloadable documents, and
specifications.

## Product Fields

SKU, product name, slug, short/long description, CAS number, molecular
formula, molecular weight, purity, sequence, storage, solubility,
appearance, research-use statement, availability, minimum order quantity,
wholesale/distributor/retail price, featured flag, status, visibility,
sort order.

## Category Domain

Tables: `categories`, `category_images`, `category_banners`,
`category_translations` (future). Unlimited nesting, parent-child
relationships, featured categories, icons, descriptions.

## Media Domain

Tables: `media`, `media_folders`, `media_tags`. Storage buckets: products,
categories, coa, msds, hplc, blog, research, invoices, avatars, banners,
general.

## Document Domain

Tables: `documents`, `coa_documents`, `msds_documents`, `hplc_documents`,
`technical_documents`, `research_documents`, `certificate_documents`. Each
document supports version, language, visibility, download counter,
approval status, file metadata.

> As in prior volumes: this schema stores/versions/serves these documents.
> Their actual content is authored from the user's own lab/compliance
> sources, not generated by Claude.

## Shopping Cart Domain

Tables: `shopping_carts`, `shopping_cart_items`, `saved_carts`,
`guest_carts`. Persistent carts survive login/logout.

## Order Domain

Tables: `orders`, `order_items`, `order_status_history`, `order_notes`,
`order_documents`, `shipping_details`, `payment_details`,
`tracking_events`. Each order supports multiple products, multiple
shipments, invoice, documents, tracking, internal notes.

## Quote Domain

Tables: `quotes`, `quote_items`, `quote_messages`,
`quote_status_history`, `quote_documents`. Supports expiry, approval,
negotiation, conversion to order.

## Invoice Domain

Tables: `invoices`, `invoice_items`, `invoice_payments`,
`invoice_documents`. Supports PDF generation, payment tracking, tax
fields, currency, status.

## Payment Domain

Tables: `payment_methods`, `payment_transactions`, `payment_status`,
`bank_transfer_records`, `crypto_transactions`. Future payment providers
should slot in without schema changes.

## Shipping Domain

Tables: `shipping_methods`, `shipping_rates`, `shipments`,
`shipment_tracking`, `shipment_events`, `shipping_addresses`. Carriers:
FedEx, UPS, DHL, EMS, TNT, Aramex, China Post, freight forwarders.

## Download Domain

Tables: `downloads`, `download_logs`, `download_permissions`,
`download_tokens`. Track who downloaded, when, which file, IP address,
download count.

## Blog Domain

Tables: `blog_posts`, `blog_categories`, `blog_tags`, `blog_authors`,
`blog_comments`, `blog_featured`. Drafts, publishing, scheduling.

## Research Domain

Tables: `research_articles`, `research_categories`, `research_tags`,
`research_downloads`, `research_authors`.

## FAQ Domain

Tables: `faq_categories`, `faq_questions`, `faq_votes`, `faq_feedback`.

## Support Domain

Tables: `support_tickets`, `support_messages`, `support_attachments`,
`support_categories`, `support_priorities`, `support_status`.

## Message Domain

Tables: `conversations`, `conversation_members`, `messages`,
`message_attachments`, `message_reads`.

## Newsletter Domain

Tables: `newsletter_subscribers`, `newsletter_lists`,
`newsletter_campaigns`, `newsletter_logs`.

## Analytics Domain

Tables: `page_views`, `search_history`, `product_views`, `cart_events`,
`order_statistics`, `download_statistics`, `visitor_sessions`. Track
referrers, countries, devices, browsers.

## Settings Domain

Tables: `system_settings`, `email_templates`, `site_settings`,
`company_information`, `shipping_settings`, `invoice_settings`,
`notification_settings`.

## Search Indexes

Index: product name, SKU, slug, CAS number, category, status, visibility,
customer email, order number, invoice number, download ID, created date.
Optimize for search performance.

## Foreign Key Rules

`ON UPDATE CASCADE`, `ON DELETE RESTRICT`. Prefer soft deletes over hard
deletes.

## Data Lifecycle

**Product:** Draft → Pending Review → Published → Archived → Deleted.
**Order:** Pending → Confirmed → Processing → Packed → Shipped →
Delivered → Cancelled → Returned.

## Audit Trail

Maintain audit records for products, orders, invoices, customers,
documents, settings, role changes, user actions. Never permanently lose
business-critical history.

## Storage Structure

Supabase Storage buckets: `/products`, `/categories`, `/documents`,
`/coa`, `/msds`, `/hplc`, `/blog`, `/research`, `/invoices`, `/media`,
`/avatars`, `/banners`. Organize files by year/month where appropriate.

## Row Level Security (RLS)

Design policies for: authenticated users, customers, administrators,
editors, support staff, distributors, guests. Users can only access their
own customer records unless they have elevated privileges.

## Prisma Organization

Structure the schema into logical sections. Use enums, relations, indexes,
unique constraints, cascade rules, named relations. Document complex
relationships with comments.

## Database Migrations

Every schema change: create a migration, include a rollback strategy,
update the Prisma Client, maintain backward compatibility where possible.
Never modify production tables manually.

## Seeding Strategy

Seed: roles, permissions, categories, sample products, research articles,
FAQs, shipping methods, company settings, admin user, demo customer.
Seeds must be idempotent.

## Performance Optimization

Indexes, partial indexes, composite indexes, query analysis, pagination,
connection pooling. Avoid N+1 queries.

## Backups

Daily backups, point-in-time recovery, migration snapshots, database
export, documented backup procedures.

## Data Retention

Configurable retention policies for logs, sessions, downloads, messages,
analytics, temporary files.

## Database Documentation

Every model documents: purpose, relationships, indexes, constraints,
business rules, usage examples, migration notes.

## Database Quality Checklist

UUIDs used, foreign keys defined, indexes optimized, relations documented,
soft deletes supported, audit fields included, Prisma models validated,
RLS policies designed, storage buckets defined, seed data prepared,
migration strategy documented, future expansion considered.

## Database Mode

When working in Database Mode:

1. Analyse the requested data requirements.
2. Design or extend the schema.
3. Update Prisma models.
4. Generate SQL migrations where needed.
5. Define indexes and constraints.
6. Specify RLS policies.
7. Explain data relationships.
8. Include seed data if appropriate.
9. Provide testing guidance for the schema.
10. Ensure compatibility with existing modules and APIs.

## Future Extensibility

Accommodate without redesign: multi-language content, multi-currency
pricing, multi-warehouse inventory, supplier management, purchase orders,
LIMS, CRM, ERP integrations, AI recommendation engines, mobile
applications, regional pricing, tax jurisdictions. Extensions should build
on existing relationships rather than introducing parallel data
structures.
